# /site-hearing

ユーザーにヒアリングを行い、サイトの仕様書を作成する。

## 引数
なし

## 処理フロー

### 1. ヒアリング

以下の項目について、対話形式でユーザーにヒアリングする。
一度にすべてを聞かず、回答に応じて深掘りしながら進める。

#### サイトの目的・概要
- サイトの目的（コーポレートサイト、ポートフォリオ、ランディングページ等）
- ターゲットユーザー
- サイトで伝えたいこと・達成したいこと

#### ページ構成
- 必要なページの一覧
- 各ページの役割と掲載する内容
- ページ間の導線

#### デザインの方向性
- 参考サイトやイメージ（あれば）
- トーン・雰囲気（モダン、クラシック、ポップ、ミニマル等）
- カラーの好み（あれば）
- フォントの好み（あれば）

#### コンテンツ
- テキストコンテンツの有無（ユーザーが用意するか、仮テキストでよいか）
- 画像・動画の扱い（プレイスホルダーで仮置きする旨を説明）

#### インタラクション
- アニメーションの希望（スクロールアニメーション、ホバーエフェクト等）
- 動的な要素の有無（スライダー、アコーディオン、モーダル等）

#### 技術的な要件
- レスポンシブ対応（必須であることを説明）
- ブラウザ対応範囲
- SEO要件
- アクセシビリティ要件

### 2. 仕様書の作成

ヒアリング内容をもとに `docs/specification.md` を作成する。

#### 仕様書の構成
```markdown
# サイト仕様書

## 1. 概要
- サイトの目的
- ターゲットユーザー
- サイトのゴール

## 2. ページ構成
- ページ一覧（ページ名、URL パス、概要）
- サイトマップ

## 3. 各ページの詳細
### ページ名
- 掲載するセクション
- 各セクションの内容
- インタラクション

## 4. デザイン方向性
- トーン・雰囲気
- カラーの方向性
- 参考イメージ

## 5. 共通要素
- ヘッダー
- フッター
- ナビゲーション
- 共通コンポーネント

## 6. 技術要件
- レスポンシブ対応
- ブラウザ対応
- SEO
- アクセシビリティ
```

### 3. ユーザー確認

- 作成した仕様書をユーザーに提示する
- 修正点があれば反映する
- 確定したら、次のステップ `/site-design` を案内する
